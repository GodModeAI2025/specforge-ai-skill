Was schmiedet SpecForge?

Vom vagen Feature-Wunsch zum wasserdichten Bauplan

Das Problem ohne SpecForge

Stell dir vor, du sagst einem KI-Tool: "Bau mir eine Authentifizierung." Was passiert? Es schreibt sofort Code — ohne zu fragen, welche Sicherheitsstandards gelten, ob es MFA braucht, oder was passiert wenn jemand 100 Mal das falsche Passwort eingibt.

Das Ergebnis: Code der "funktioniert" — aber Sicherheitslücken hat, keine klaren Anforderungen, und niemand weiß ob er vollständig ist.

💡
Die Kernidee

SpecForge dreht den Prozess um: Erst spezifizieren, dann bauen. "Governance ist ein Compiler, kein Komitee" — Regeln werden automatisch geprüft, nicht in Meetings diskutiert.

Das F-Stufen-System

Jeder Check an einem Gate hat einen Schweregrad — die F-Stufen. Je höher die Stufe, desto ernster der Befund. Denk an eine Ampel: Grün (weiter), Gelb (Achtung), Rot (Stopp).

F0 PASS — Kein Mangel. Alles in Ordnung, weiter geht's.
F1 INFO — Kleinigkeit. Empfehlung notiert, Gate passierbar.
F2 WARNING — Mittelgewicht. Gate passierbar, aber Pflicht-Task vor Go-Live.
F3 CONDITIONAL — Gewichtig. Nur weiter mit dokumentierter Risiko-Akzeptanz.
F4 FAIL — Schwer. Gate blockiert. Muss behoben werden.

So sieht ein Gate-Check aus

Hier ein echtes Beispiel wie SpecForge einen Gate-Check ausgibt:

GATE OUTPUT 
── Gate G4: Analyze → Implement ──
✅ [F0] EARS-Formulierung: 5/5 Stories
✅ [F0] Gherkin-Szenarien: 12 (min. 10)
✅ [F0] Constitution: vorhanden
⚠️ [F3] STRIDE: 2/5 Stories offen
   └─ Risiko-Akzeptanz erforderlich
⚠️ [F2] NFR DAT-03: Verschlüsselung
   └─ Pflicht-Task vor Go-Live
── Ergebnis: CONDITIONAL ──
KLARTEXT

Die Überschrift zeigt: Wir prüfen den Übergang von Analyze zu Implement.

Drei Prüfpunkte sind bestanden (F0) — alle Stories haben EARS, genug Gherkin-Tests, und die Constitution existiert.

Aber: Bei 2 von 5 Stories fehlt noch die STRIDE-Sicherheitsanalyse. Das ist ein F3 — weiter nur mit Chef-Unterschrift.

Außerdem fehlt die Transport-Verschlüsselung (F2) — weiter geht's, aber ein Fix-Task muss vor dem Launch erledigt werden.

Ergebnis: CONDITIONAL — das Gate ist nicht blockiert, aber jemand muss das Risiko formal akzeptieren.

🧠
Warum das wichtig ist

Gate-Checks verhindern dass Probleme sich aufstauen. Ohne sie merkst du erst beim Launch dass etwas fehlt — wenn es 10x teurer ist es zu fixen.

04

Die Sprache der Specs

Warum "soll schnell laden" nie in einer Spec stehen darf

EARS — 5 Satzbaupläne für Anforderungen

SpecForge benutzt EARS-Syntax: Feste Satzmuster die dafür sorgen, dass jede Anforderung eindeutig und testbar ist. Wie Grammatikregeln, aber für Software-Anforderungen.

🌍 Ubiquitous
Gilt immer. "Das System soll alle Responses im JSON-Format ausliefern."
⚡ Event-Driven
Wenn [Ereignis], soll das System [Reaktion]. Ausgelöst durch ein Ereignis.
🔄 State-Driven
Solange [Zustand], soll das System [Verhalten]. Gilt während eines Zustands.
⚙️ Optional
Wenn [Konfiguration], soll das System [Funktion]. Nur wenn aktiviert.
🛡️ Unwanted
Wenn [Fehler/Angriff], soll das System [Schutzmaßnahme]. Für Fehlerfälle.

Verbotene Wörter

SpecForge hat eine Blocklist — Wörter die in einer Spec nichts verloren haben, weil sie vage und nicht testbar sind:

"schnell" → "≤200ms (p95)" — messbar, testbar
"viele" → "≥10.000 gleichzeitige Verbindungen"
"sicher" → "AES-256-GCM Verschlüsselung nach BSI TR-02102"
"einfach" → "≤3 Klicks bis zur Zielaktion"
"in Echtzeit" → "≤100ms Latenz"
⚠️
Anti-Pattern AP-04

Wenn eines dieser Wörter in einer Story auftaucht, ist das ein F4-Befund (FAIL). Das Gate blockiert — die Story muss überarbeitet werden bis sie messbar ist.

Gherkin — Tests als Geschichten

Jede Story bekommt mindestens 2 Gherkin-Szenarien: Einen Happy Path (alles geht gut) und einen Fehlerfall (was wenn etwas schiefgeht).

GHERKIN 
Szenario: Erfolgreicher Login mit MFA
  Gegeben ein registrierter Benutzer
    und TOTP ist aktiviert
  Wenn er E-Mail + Passwort eingibt
    und den korrekten TOTP-Code
  Dann wird eine Session erstellt
    und er sieht das Dashboard

Szenario: Login mit falschem TOTP
  Gegeben ein registrierter Benutzer
  Wenn er einen falschen TOTP-Code eingibt
  Dann wird der Zugang verweigert
    und der Vorfall wird geloggt
KLARTEXT

Happy Path: Wenn alles richtig läuft — Benutzer gibt alle Daten korrekt ein und bekommt Zugang.

"Gegeben" beschreibt die Ausgangssituation: Es gibt einen Benutzer, und der hat MFA aktiviert.

"Wenn" beschreibt die Aktion: Was der Benutzer tut.

"Dann" beschreibt das erwartete Ergebnis: Was das System tun soll.

Fehlerfall: Was passiert wenn etwas schiefgeht — falscher Code → Zugang verweigert + Security-Log.

Mindestens 2 Szenarien pro Story (AP-06). Sonst fehlt die Absicherung gegen Fehler.

05

Die Wächter

10 Golden Principles, 8 Anti-Patterns — und eine Enforcement Engine

Die 10 Gebote — Golden Principles

SpecForge hat 10 unverletzliche Regeln — die Golden Principles. Sie sind wie die Grundgesetze deines Projekts — keine Verhandlungssache. Hier die wichtigsten:

2
GP-02: Spec-before-Code

Keine einzige Zeile Code ohne passenden Spec-Eintrag. Wie ein Baugenehmigung — ohne Plan kein Bau.

3
GP-03: ADR-Disziplin

Jede wichtige technische Entscheidung wird in einem ADR dokumentiert — damit man in 6 Monaten noch weiß WARUM.

8
GP-08: Prinzip-Unverletzlichkeit

Das Meta-Prinzip: GP-Verstöße blockieren ALLES bis sie behoben sind. Kein Wegschauen möglich.

💡
Governance als Compiler

In der Softwareentwicklung ist ein Compiler ein Programm das deinen Code prüft und dir sofort sagt was falsch ist — bevor du es ausführst. SpecForge macht dasselbe mit deinen Anforderungen: Es prüft automatisch und blockiert bei Verstößen. Kein Komitee das entscheidet, sondern eine Maschine die durchsetzt.

Anti-Patterns — Die 8 Todsünden

SpecForge kennt 8 häufige Fehler in Anforderungen und erkennt sie automatisch:

AP-01 · Implementation Bias
WIE statt WAS beschreiben. "Benutze Redis für Caching" → soll heißen WAS gecacht wird.
AP-04 · Vage Quantifizierung
"Schnell", "sicher", "viele" — alles nicht messbar. Sofort F4 (FAIL).
AP-05 · Scope Creep
Tasks die keinen Bezug zur Spec haben. Code ohne Auftrag = außer Kontrolle.
AP-06 · Missing Negative
Nur Happy Path, keine Fehlerfälle. Wenn du nicht planst was schiefgehen kann, WIRD es schiefgehen.
06

STRIDE — Das Sicherheits-Röntgen

6 Bedrohungskategorien die SpecForge bei jeder sicherheitsrelevanten Story prüft

Was ist STRIDE?

Stell dir vor, du baust ein Haus. Ein Sicherheitsexperte geht durch und prüft: Kann man die Tür aufbrechen? Kann jemand durchs Fenster klettern? Kann man die Alarmanlage umgehen? STRIDE macht dasselbe für Software — nur systematischer.

🎭
S · Spoofing
Kann sich jemand als ein anderer ausgeben? Wie ein gefälschter Ausweis an der Tür.
✏️
T · Tampering
Kann jemand Daten unterwegs verändern? Wie ein Brief der auf dem Postweg geöffnet wird.
🙈
R · Repudiation
Kann jemand abstreiten was er getan hat? Wie eine Transaktion ohne Quittung.
🔓
I · Info Disclosure
Können vertrauliche Daten nach außen gelangen? Wie ein offenes Fenster zum Tresorraum.
🔌
D · Denial of Service
Kann jemand das System lahmlegen? Wie wenn 10.000 Leute gleichzeitig durch eine Drehtür wollen.
👑
E · Elevation of Privilege
Kann sich ein normaler Nutzer Admin-Rechte verschaffen? Wie ein Praktikant der den Tresorschlüssel findet.

Wie streng ist STRIDE?

Das hängt vom Profil ab — SpecForge passt die Anforderungen ans Risiko an:

K
KRITIS: Alle 6 Kategorien Pflicht für JEDE Story.

F4 (FAIL) wenn STRIDE fehlt. Kein Durchkommen ohne vollständige Analyse.

S
Standard: Pflicht für sicherheitsrelevante Stories.

F3 (CONDITIONAL) wenn es fehlt. Weiter nur mit Risiko-Akzeptanz.

U
Startup: Optional, empfohlen bei Auth/Payment.

F1 (INFO) wenn es fehlt. Nur eine Empfehlung.

So sieht eine STRIDE-Bewertung aus

STRIDE OUTPUT 
#### STRIDE: SF-SEC-001 (MFA-Login)

| Kategorie        | Rel. | Risiko  | Status    |
| Spoofing         | Ja   | Hoch   | Mitigiert |
| Tampering        | Ja   | Mittel | Mitigiert |
| Repudiation      | Ja   | Mittel | Mitigiert |
| Info Disclosure  | Ja   | Hoch   | Offen     |
| Denial of Service| Ja   | Mittel | Mitigiert |
| Elev. of Priv.   | Ja   | Niedrig| Mitigiert |

Offene Risiken: 1
Höchstes Risiko: Info Disclosure (Hoch)
KLARTEXT

Jede der 6 STRIDE-Kategorien wird einzeln bewertet: Ist sie relevant? Wie hoch das Risiko? Ist es behoben?

Spoofing (Hoch, Mitigiert): Identitätstäuschung ist relevant — aber durch MFA + JWT-Validierung abgesichert.

Info Disclosure (Hoch, Offen): Es fehlt noch eine Maßnahme gegen Informationsoffenlegung. Das ist ein offener Befund!

Ein offenes Risiko bei "Hoch" wird zum F3-Befund im nächsten Gate — das heißt: CONDITIONAL, jemand muss das Risiko akzeptieren oder es muss gefixt werden.

💡
Defense in Depth

STRIDE zwingt dich, an Angriffe zu denken BEVOR du Code schreibst. In der Sicherheitswelt heißt das "Defense in Depth" — mehrere Schutzschichten, wie eine Burg mit Graben, Mauer und Wachtürmen. Keine einzelne Maßnahme reicht allein.

Das Nervensystem — specforge.json

Eine einzige Datei steuert das gesamte Verhalten

specforge.json — Das Gehirn

Jedes SpecForge-Projekt hat eine zentrale Steuerdatei. Stell sie dir vor wie die DNA eines Lebewesens — sie bestimmt wie streng geprüft wird, welche Regeln gelten, und wo die Artefakte landen. Wie die Einstellungen-App auf deinem Handy, nur für dein ganzes Projekt.

SPECFORGE.JSON 
{
  "project": "Kundenportal",
  "profile": "Standard",
  "stack": {
    "framework": "Next.js",
    "language": "TypeScript",
    "database": "PostgreSQL"
  },
  "regulations": ["DSGVO"],
  "active_gps": [1,2,3,4,5,6,7,8],
  "audit": true
}
KLARTEXT

project: Wie heißt das Projekt? Wird in allen Artefakten als Titel verwendet.

profile: Standard = mittlere Strenge. Bestimmt welche Checks Pflicht sind und welche optional.

stack: Die Technologien. SpecForge nutzt das um den Plan und die Tasks framework-spezifisch zu generieren.

regulations: Welche Gesetze gelten? DSGVO aktiviert automatisch Datenschutz-Checks (DAT-01 bis DAT-07).

active_gps: Welche der 10 Golden Principles sind aktiv? Standard hat GP 1–8, KRITIS alle 10.

audit: Soll ein Audit Trail geführt werden? Bei KRITIS immer true.

09

Execution & Handoff

Enterprise-Ready Execution: SpecForge 🤝 GSD & MissionForge

Die Spec ist fertig — was nun?

SpecForge generiert audit-sichere Specs (Phase 1). Um diese Anforderungen nun in echten Code oder Aktionen umzuwandeln (Phase 2), gibt es zwei hochspezialisierte Ausführungspfade, die nahtlos in das Enterprise Skill Governance Framework integrieren.

Weg A: Terminal-Execution

Get Shit Done (GSD)

Ideal für massive Code-Änderungen. GSD läuft in der CLI (z.B. Claude Code), konsumiert die spec.md und arbeitet sie in isolierten 200k-Token-Subagenten ab.

  • ✅ Parallele Agenten-Ausführung
  • ✅ Saubere, atomare Git-Commits
  • ✅ Kein Context Rot bei großen Features
Weg B: In-Chat-Execution

MissionForge & TaskPulse

Ideal für komplexe Workflows direkt im Chat. Spawnt eine ganze Agenten-Company (Planung, Ausführung, QS). Jeder Sub-Agent erhält spezifische Tasks basierend auf den REQ-IDs aus SpecForge.

  • ✅ Komplett im Chat-Interface
  • ✅ Nutzt TaskPulse-Protokolle (CMDB/ITIL konform)
  • ✅ Hierarchische Agenten-Delegation
📋
ITIL & Governance Compliance

Sowohl GSD als auch MissionForge hinterlassen standardisierte Ausführungsprotokolle (z.B. via TaskPulse-Integration). So erfüllt die gesamte Wertschöpfungskette (vom Jira-Ticket zur Spec bis zum Git-Commit) die strengsten Auditing-Anforderungen des Skill Governance Frameworks.

Specs schmieden, nicht schreiben

SpecForge heißt nicht umsonst "Forge" — eine Schmiede. Gute Spezifikationen entstehen nicht durch Tippen, sondern durch Erhitzen (Analyse), Hämmern (Clarify), Formen (Plan) und Härten (Enforce). Jetzt kennst du die Werkzeuge.
